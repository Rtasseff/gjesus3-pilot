# MFB gjesus3 RDM Pilot – Updated To‑Do List (post-team sync)

## Context captured

* Four storage areas:

  * **Staging**: very temporary, no imposed structure/documentation.
  * **Raw**: rigid + long-term; structured by instrument/modality and date; **one primary raw data file per acquisition folder**.
  * **Publication**: rigid + long-term at top level; per-publication subfolders with flexible internal structure until “closed.”
  * **Project**: temporary working space but semi-ordered and documented.
* Registries:

  * **Top-level registries** (one place): raw registry, publication registry, project registry.
  * **Local provenance logs**: live *inside each* publication folder and each project folder (created at creation time).

## C) Define the three top-level registries (schemas + ownership)

### C1. Raw registry (top-level)

* [ ] Decide schema (one row per acquisition folder / one primary raw file).

  * Required fields (draft):

    * `RawID/AcqID`, `instrument`, `date_time forcing`, `operator`, `primary_file_name`, `file_ext`, `raw_path`, `checksum_manifest_present`, `project_hint (optional)`, `sample/animal pointers (optional if sensitive)`
* [ ] Decide where “extended metadata” lives (sidecar file path field vs embedded columns).
* [ ] Define update rules (who edits, when edits are allowed, how corrections are logged).

### C2. Publication registry (top-level)

* [ ] Decide a **publication ID scheme** (e.g., `PUB-0001`) + **folder-safe short name** rules.
* [ ] Decide minimal fields:

  * `PubID`, `short_name`, `working_title`, `status` (in prep/submitted/published), `PI`, `first_author`, `corresponding_author`, `journal (optional)`, `DOI (when available)`, `created_date`, `closed_date (optional)`
* [ ] Explicitly confirm: **do not require listing all RawIDs in the publication registry** (linkage via provenance/logs).
* [ ] Define the “closure step” for publications (final title + DOI + status update).

### C3. Project registry (top-level)

* [ ] Define a **project ID scheme** (e.g., `PROJ-0001`) + folder-safe short name rules.
* [ ] Decide minimal fields:

  * `ProjectID`, `short_name`, `description`, `SPOC/owner`, `start_date`, `status` (active/paused/closed), `notes (optional)`
* [ ] Define what “a project” means for this pilot (even if fuzzy): expected use cases and anti-patterns.

## D) Provenance logging (local per project + per publication)

### D1. Decide the format and template

* [ ] Choose the local provenance filename and format (e.g., `provenance.tsv` or `provenance.csv`).
* [ ] Define template fields (as agreed):

  * Required: `fileID`, `name` (stem), `file_type` (ext), `date`, `creator`, `input_refs (RawIDs/FileIDs)`, `output_path`, `process_description`
  * Optional: `lab_notebook_ref`, `software+version`, `parameters_ref`
* [ ] Confirm the unit of record: **one row per tangible output file** (inputs/process → single output).

  * If a process yields multiple outputs, record multiple rows with the same inputs/process.

### D2. Decide minimum provenance rules (pilot)

* [ ] Publication folder: **log all non-predefined files** (default yes; this is long-term).
* [ ] Project folder: define what must be logged (recommended minimum):

  * [ ] any file intended for publication export
  * [ ] any conversion (e.g., vendor → OME-TIFF)
  * [ ] any shared dataset (internal/external)
* [ ] Define how provenance logs relate to closure:

  * [ ] publication “closed” requires provenance file present + minimally complete
  * [ ] project “closed” requires provenance file present or a closure note explaining what is kept
