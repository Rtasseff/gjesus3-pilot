## 1. Purpose and Guiding Principles

This document establishes the storage, organization, and governance rules for microscopy data generated by the MFB group. Compliance is mandatory for all users granted access to gjesus3.

**Core principles:**

1. **Raw data is sacred.** Original acquisition files are never modified, renamed, or moved after registration.
2. **Traceability is non-negotiable.** Every derived output must be traceable back to its raw source(s).
3. **Structure enables collaboration.** Consistent conventions allow any group member to find, interpret, and build upon existing work.
4. **Lightweight now, scalable later.** The system must be implementable immediately and compatible with future platform/OMERO integration.

## 2. Directory Structure

The top-level structure on `gjesus3` separates immutable raw data from active project work:

```
/gjesus3/microscopy/
│
├── raw/                                    # Authoritative raw acquisitions (READ-ONLY after deposit)
│   ├── registry_raw_inventory.csv          # Master list of all raw acquisitions
│   ├── zeiss-wsi/                          # Instrument-specific subfolder
│   │   ├── 2025/                           # Year of acquisition
│   │   │   ├── 2025-01/                    # Month of acquisition
│   │   │   │   ├── <scan_id>/              # One folder per scan session
│   │   │   │   │   ├── *.czi               # Raw image file(s)
│   │   │   │   │   ├── *.xml               # Instrument metadata (if separate)
│   │   │   │   │   └── README.txt          # Acquisition notes (see Section 4)
│   │   │   │   └── ...
│   │   │   └── ...
│   │   └── ...
│   └── other-instruments/                  # Future expansion (histology, etc.)
│
├── projects/                               # Active project workspaces (READ-WRITE)
│   ├── registry_projects.csv               # Registered projects
│   ├── <project_code>/                     # One folder per registered project
│   │   ├── _project_metadata.[yaml|json]   # Project metadata file (required) — format TBD
│   │   ├── registry_provenance_ledger.csv  # Provenance tracking for this project
│   │   ├── raw_linked/                     # Symlinks to raw data (if automated deposit used)
│   │   ├── [subdirectories TBD]            # Internal project organization — see discussion below
│   │   └── ...
│   └── ...
│
└── _templates/                             # Templates and reference documents
    ├── project_template/                   # Starter structure for new projects
    ├── _project_metadata.template          # Project metadata template (format TBD)
    └── README_acquisition.txt              # Template for acquisition notes
```

### 2.1 Key Structural Decisions

**Registry placement:** Rather than a central `_registry/` folder, each registry lives at the top of its relevant branch:
- `raw/registry_raw_inventory.csv`
- `projects/registry_projects.csv`
- `projects/<project_code>/registry_provenance_ledger.csv`

> **🔶 DISCUSSION NEEDED: Project Internal Organization**
> 
> How should projects be subdivided internally? Options include:
> - By researcher (e.g., `projects/mfb-lung-study/mbc/`, `projects/mfb-lung-study/iff/`)
> - By study/experiment (e.g., `projects/mfb-lung-study/study-01/`, `projects/mfb-lung-study/study-02/`)
> - Flat structure with no required subdivisions (researcher discretion)
> - ISA-style hierarchy (see Section 2.2)
> 
> The previous draft included `inputs/`, `processing/`, and `outputs/` subdirectories. These have been removed because:
> - Outputs can become inputs for subsequent processes
> - Enforcing symlinks or path references in these folders adds complexity for non-technical users
> - The provenance ledger already captures input/output relationships
> 
> **Need team input on what level of internal structure is helpful vs. burdensome.**

### 2.2 Alignment with Standards: ISA Model Consideration

> **🔶 DISCUSSION NEEDED: ISA Framework Adoption**
> 
> The ISA (Investigation-Study-Assay) model ([ISA specifications](https://isa-specs.readthedocs.io/en/latest/isamodel.html)) provides a well-established standard for organizing experimental metadata. A potential mapping:
> 
> | ISA Concept | MFB Mapping |
> |-------------|-------------|
> | **Investigation** | What we're calling "Project" |
> | **Study** | Potential subdivision within a project (subject, design and contextual info for 1+ assays) |
> | **Assay** | Roughly equivalent to a raw acquisition/scan session |
> 
> **Advantages of ISA alignment:**
> - Established standard with community tools
> - Facilitates data sharing and publication
> - Metadata schemas already defined
> - Compatible with OMERO and other platforms
> 
> **Concerns:**
> - ISA has many fields — may be too heavy to start with
> - Terminology shift (Investigation vs. Project) may confuse team
> - Requires JSON or tabular metadata formats (not YAML)
> 
> **Options:**
> 1. **Full ISA adoption:** Rename "projects" to "investigations," use ISA JSON/CSV formats
> 2. **ISA-lite:** Use ISA terminology and structure but only require a subset of fields
> 3. **ISA-compatible:** Keep current naming but ensure metadata can be exported to ISA format later
> 4. **Defer:** Start with simple structure, migrate to ISA when platform requires it
> 
> **Need team input: Is the overhead of ISA worth the standardization benefits? Which option?**

## 4. Raw Data Deposit Workflow

### 4.1 Overview

The deposit workflow moves data from the acquisition instrument to the permanent `raw/` storage on gjesus3.

> **🔶 DISCUSSION NEEDED: Direct Save vs. Staging Area**
> 
> **Options under investigation:**
> 
> 1. **Direct network save:** Configure instrument to save directly to `gjesus3/microscopy/raw/zeiss-wsi/...`
>    - Requires network configuration and testing
>    - Risk: If naming convention not followed at acquisition time, raw folder gets messy
> 
> 2. **Local save + manual transfer:** Operator saves to local drive, then copies to gjesus3
>    - Current assumed workflow
>    - Risk: Delays, forgotten transfers, inconsistent organization
> 
> 3. **Staging area + automated migration:** Save to temporary staging folder with flexible naming, then use a script to migrate to final location
>    - Script could enforce metadata collection
>    - Script could auto-read instrument metadata
>    - Script could create raw folder, project folder, and symlinks
>    - More robust but requires script development
> 
> **Need to investigate:**
> - Can we set up network save to gjesus3 from the Zeiss workstation?
> - What does the platform want to do with initial raw scans (if anything)?
> - Is script development feasible in the initial rollout?
> 
> **Need IT consultation and platform coordination before finalizing.**

### 4.2 Pre-Acquisition

1. **Plan the session.** Know which project(s) the data will support before scanning.
2. **Verify access.** Confirm you have write access to the appropriate area on gjesus3.

### 4.3 Acquisition

1. **Follow instrument protocols.** Use standard Zeiss acquisition settings; record any deviations.
2. **Save to designated location** (per workflow option selected above).

### 4.4 Deposit to gjesus3 (Manual Workflow)

*This section applies if Option 2 (manual transfer) is used. If automated staging is implemented, this will be replaced by script instructions.*

1. **Create the scan folder:**
   ```bash
   mkdir -p /gjesus3/microscopy/raw/zeiss-wsi/2025/2025-01/20250122_ZWSI_MBC_001/
   ```

2. **Copy (not move) raw files** into the scan folder. Verify copy integrity.

3. **Add acquisition metadata** — see Section 4.5.

4. **Generate checksums:**
   ```bash
   cd /gjesus3/microscopy/raw/zeiss-wsi/2025/2025-01/20250122_ZWSI_MBC_001/
   sha256sum *.czi > checksums.sha256
   ```

5. **Register in the raw inventory** — add entry to `raw/registry_raw_inventory.csv` (see Section 6.1).

6. **Set folder permissions to read-only** (Data Management Lead responsibility):
   ```bash
   chmod -R a-w /gjesus3/microscopy/raw/zeiss-wsi/2025/2025-01/20250122_ZWSI_MBC_001/
   ```

**Timeline:** Registration must be completed **same day** as acquisition.

2. **Create the project folder** using the template:

         /gjesus3/microscopy/projects/<project_code>/
   ```

3. **Complete project metadata file** with required fields (format TBD — see Section 6.3).

### 5.2 Linking Raw Data to Projects

> **🔶 DISCUSSION NEEDED: Raw Data Linking Approach**
> 
> **Purpose:** Enable projects to reference their source raw data without duplicating files.
> 
> **Options:**
> 
> 1. **Automated symlinks via deposit script:**
>    - If staging + script workflow is implemented (Section 4.1 Option 3)
>    - Script creates symlinks in `projects/<project_code>/raw_linked/`
>    - Minimal researcher effort
>    - **Recommended if automation is feasible**
> 
> 2. **Manual symlinks:**
>    ```bash
>    ln -s /gjesus3/microscopy/raw/zeiss-wsi/2025/2025-01/20250122_ZWSI_MBC_001/ \

>    - Requires command-line comfort
>    - Risk of errors, broken links, researcher reluctance
> 
> 3. **Path references only (no symlinks):**
>    - Just record scan IDs in provenance ledger
>    - No actual links in file system
>    - Simpler but less convenient for analysis workflows
> 
> **Key question for researchers:**
> - How comfortable are you with creating symlinks?
> - Would you use them if they existed, or prefer to navigate to raw/ directly?
> - Would automated linking at deposit time be valuable?
> 
> **Recommendation:** If we implement automated deposit (Section 4.1 Option 3), include symlink creation. Otherwise, rely on provenance ledger references only and skip the `raw_linked/` folder.
> 
> **Need researcher input on workflow preferences.**

### 5.3 Recording Provenance

> **🔶 DISCUSSION NEEDED: Provenance Recording — Critical for Traceability**
> 
> **Purpose:** Track how derived outputs were created from raw inputs, enabling trace-back from any publication figure to its source data.
> 
> **Importance:** This is essential for publication traceability and reproducibility. However, it is only useful if researchers actually do it.
> 
> **Key challenges:**
> 
> 1. **Multiple inputs:**
>    - A derived file may have many sources: multiple raw scans, parameter files, reference images, etc.
>    - Need a format that handles variable numbers of inputs
>    - Proposal: separate multiple paths with `;` in a single field
> 
> 2. **Multiple outputs:**
>    - Easier: one row per output, repeating inputs and process description
> 
> 3. **Process description:**
>    - What level of detail is needed?
>    - Options: free text, script name, git repo + commit, ELN entry ID, paper notebook date
>    - Need to balance flexibility, machine-readability, and actual traceability
> 
> 4. **Researcher compliance:**
>    - If this is too burdensome, researchers won't do it
>    - If it's not done, raw data tracking becomes impossible
> 
> **Proposed approach:**
> 
> | Field | Required | Format | Notes |
> |-------|----------|--------|-------|
> | `output_path` | Yes | Relative path within project | e.g., `analysis/fig1_panel_a.png` |
> | `input_sources` | Yes | Scan IDs and/or file paths, separated by `;` | e.g., `20250122_ZWSI_MBC_001;params/config.yaml` |
> | `process_reference` | Yes | Free text — but encourage specificity | e.g., `segment_cells.py v1.2`, `ELN-2025-0042`, `notebook 2025-01-22 p.15` |
> | `software` | Recommended | Name and version | e.g., `ImageJ 1.54f`, `Python 3.11 + scikit-image 0.21` |
> | `operator` | Yes | Operator code | Who created this output |
> | `date_created` | Yes | YYYY-MM-DD | When created |
> | `notes` | Optional | Free text | Any additional context |
> 
> **Questions for discussion:**
> - Is this level of detail feasible for routine work?
> - What's the minimum viable provenance record?
> - Should we require provenance only for "publication-quality" outputs, or for everything?
> - Would a simple form/script make this easier than editing CSV directly?
> 
> **Need researcher input on willingness and workflow integration.**

## 8. Access Control and Governance

### 8.1 Access Tiers

| Tier | Permissions | Users |
|------|-------------|-------|
| **Admin** | Full read/write everywhere; manages registries and permissions | Ryan Tasseff, Jesús Ruiz-Cabello |
| **Operator** | Write to deposit area; read/write to `projects/`; read-only to registered `raw/` | All authorized users |
| **Read-only** | Read access to specified projects only | TBD (external collaborators if needed) |

### 8.2 Permission Enforcement

Raw data protection is enforced via **file system permissions**:

- After registration, scan folders in `raw/` are set to read-only (`chmod -R a-w`)
- Only Admin tier can modify permissions
- Operators can create new scan folders but cannot modify registered ones

### 8.3 Compliance Requirements

Access to gjesus3/microscopy is conditional on agreement to:

1. **Follow all naming conventions** exactly as specified.
2. **Register all raw acquisitions** in the raw inventory **same day** as acquisition.
3. **Never modify, rename, or delete** files in `raw/` after registration.
4. **Record provenance** for derived outputs intended for publication or sharing.
5. **Report issues** immediately to the Data Management Lead.

### 8.4 Violation Handling

1. **First violation:** Written reminder and correction of the issue.
2. **Repeated violations:** Escalation to PI; potential temporary suspension of write access.
3. **Serious violations** (e.g., deletion of raw data): Escalation to PI; potential permanent access revocation.

### 8.5 Onboarding Process

New users must:

1. Read this document in full.
2. Attend orientation session with the Data Management Lead.
3. Complete a supervised test deposit.
4. Acknowledge compliance requirements in writing.

## 9. Data Protection Considerations

### 9.1 Standard Data

Most microscopy data (animal tissue, cell lines, phantoms) is **standard sensitivity** and requires no special handling beyond the conventions in this document.

### 9.2 Restricted Data (Human/Clinical)

Data derived from human subjects or clinical samples requires additional safeguards:

1. **Mark the project** as `data_sensitivity: restricted` in project metadata.
2. **No patient identifiers** in file names, folder names, or unencrypted metadata.
3. **Access limited** to named individuals approved for the study.
4. **Tania Orosco Salinas** serves as safeguard/advisor for clinical data handling.
5. **Consult institutional policies** and ethics approvals before data deposit.

## 10. Integrity and Backup

### 10.1 Checksums

All raw acquisitions must have SHA-256 checksums generated at deposit time. Periodic verification (quarterly recommended) ensures data integrity.

```bash
# Generate at deposit:
sha256sum *.czi > checksums.sha256

# Verify later:
sha256sum -c checksums.sha256
```

### 10.2 Backup Status

> **🔶 DISCUSSION NEEDED: Backup Strategy — Requires PI Decision**
> 
> **Current state:** gjesus3 is RAID-protected but **not backed up** by IT.
> 
> RAID protects against single-drive failure but not against:
> - Multiple drive failures
> - Accidental deletion
> - Ransomware/corruption
> - Physical disaster (fire, flood)
> 
> **Questions for Jesús:**
> 
> 1. **Do we want offsite backup?** If yes, what scope:

>    - Only `raw/` (irreplaceable original data)?

>

>    - How much data loss is tolerable?
