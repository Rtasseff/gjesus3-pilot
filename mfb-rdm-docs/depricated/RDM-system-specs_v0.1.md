**Version:** 0.1 (Draft)  
**Scope:** gjesus3 NAS — Zeiss WSI/Large-Area Microscopy (initial), expandable to other microscopy sources  
**Owner:** Jesús Ruiz-Cabello (PI, MFB)  
**Data Management Lead:** Ryan Tasseff (Data Office, CIC biomaGUNE)

---

## 1. Purpose and Guiding Principles

This document establishes the storage, organization, and governance rules for microscopy data generated by the MFB group. Compliance is mandatory for all users granted access to gjesus3.

**Core principles:**

1. **Raw data is sacred.** Original acquisition files are never modified, renamed, or moved after registration.
2. **Traceability is non-negotiable.** Every derived output must be traceable back to its raw source(s).
3. **Structure enables collaboration.** Consistent conventions allow any group member to find, interpret, and build upon existing work.
4. **Lightweight now, scalable later.** The system must be implementable immediately and compatible with future platform/OMERO integration.

## 2. Directory Structure

The top-level structure on `gjesus3` separates immutable raw data from active project work:

```
/gjesus3/microscopy/
├── raw/                          # Authoritative raw acquisitions (READ-ONLY after deposit)
│   ├── zeiss-wsi/                # Instrument-specific subfolder
│   │   ├── 2025/                 # Year of acquisition
│   │   │   ├── 2025-01/          # Month of acquisition
│   │   │   │   ├── <scan_id>/    # One folder per scan session
│   │   │   │   │   ├── *.czi     # Raw image file(s)
│   │   │   │   │   ├── *.xml     # Instrument metadata (if separate)
│   │   │   │   │   └── README.txt # Optional: acquisition notes
│   │   │   │   └── ...
│   │   │   └── ...
│   │   └── ...
│   └── other-instruments/        # Future expansion (histology, etc.)
│
├── projects/                     # Active project workspaces (READ-WRITE)
│   ├── <project_code>/           # One folder per registered project
│   │   ├── _project.yaml         # Project metadata file (required)
│   │   ├── inputs/               # Symlinks or references to raw data
│   │   ├── processing/           # Intermediate/working files
│   │   ├── outputs/              # Final derived outputs (figures, exports)
│   │   └── docs/                 # Protocols, analysis notes, manuscripts
│   └── ...
│
├── _registry/                    # System registries (managed, not user-editable)
│   ├── raw_inventory.csv         # Master list of all raw acquisitions
│   ├── provenance_ledger.csv     # Input → process → output relationships
│   └── projects.csv              # Registered projects
│
└── _templates/                   # Templates and reference documents
    ├── project_template/         # Starter structure for new projects
    ├── _project.yaml.template    # Project metadata template
    └── README_acquisition.txt    # Template for acquisition notes
```

## 3. Naming Conventions

### 3.1 Scan ID (Raw Acquisitions)

Each scan session receives a unique identifier following this pattern:

```
<YYYYMMDD>_<instrument>_<operator>_<sequence>
```

| Component | Description | Example |
|-----------|-------------|---------|
| `YYYYMMDD` | Date of acquisition | `20250122` |
| `instrument` | Short instrument code | `ZWSI` (Zeiss WSI) |
| `operator` | Operator initials (3 chars) | `MBC` (Marta Beraza Cabrerizo) |
| `sequence` | Daily sequence number (3 digits) | `001`, `002` |

**Example:** `20250122_ZWSI_MBC_001`

### 3.2 Operator Codes

Each authorized user is assigned a unique 3-character code:

| Name | Code |
|------|------|
| Marta Beraza Cabrerizo | MBC |
| Ainhize Urkola Arsuaga | AUA |
| Irene Fernández Folgueral | IFF |
| Claudia Beatriz Miranda Pérez de Alejo | CBM |
| Itziar Souto Riobo | ISR |
| Ekine Olaizola Bárcena | EOB |
| Tania Orosco Salinas | TOS |

### 3.3 Project Codes

Projects use a descriptive short code (max 20 characters, lowercase, hyphens allowed):

```
<pi-initials>-<short-description>
```

**Examples:** `jrc-lung-fibrosis`, `mbc-tumor-margins`, `tos-clinical-validation`

### 3.4 Derived File Naming

Derived files should indicate their source and processing step:

```
<source_scan_id>_<processing_step>_<version>.<ext>
```

**Example:** `20250122_ZWSI_MBC_001_segmented_v2.tiff`

## 4. Raw Data Deposit Workflow

### 4.1 Pre-Acquisition

1. **Plan the session.** Know which project(s) the data will support before scanning.
2. **Verify access.** Confirm you have write access to `/gjesus3/microscopy/raw/zeiss-wsi/`.

### 4.2 Acquisition

1. **Follow instrument protocols.** Use standard Zeiss acquisition settings; record any deviations.
2. **Save directly to staging area** (if instrument supports network save) or to local drive for transfer.

### 4.3 Deposit to gjesus3

1. **Create the scan folder:**
   ```bash
   mkdir -p /gjesus3/microscopy/raw/zeiss-wsi/2025/2025-01/20250122_ZWSI_MBC_001/
   ```

2. **Copy (not move) raw files** into the scan folder. Verify copy integrity.

3. **Add acquisition notes** (optional but encouraged):
   ```
   /gjesus3/microscopy/raw/zeiss-wsi/2025/2025-01/20250122_ZWSI_MBC_001/README.txt
   ```
   Include: sample description, staining, objective, any anomalies.

4. **Register in the raw inventory** (see Section 6).

5. **Lock the folder.** After registration, the scan folder becomes read-only. (Enforcement mechanism TBD — may be file permissions or policy-based initially.)

### 4.4 Integrity Verification

After deposit, generate and store a checksum:

```bash
sha256sum *.czi > checksums.sha256
```

This enables future verification that files have not been corrupted or altered.

## 5. Project Workspace Workflow

### 5.1 Creating a New Project

1. **Register the project** by adding an entry to `/gjesus3/microscopy/_registry/projects.csv`.

2. **Create the project folder** using the template:
   ```bash
   cp -r /gjesus3/microscopy/_templates/project_template/ \
         /gjesus3/microscopy/projects/<project_code>/
   ```

3. **Complete `_project.yaml`** with required metadata (see Section 7.2).

### 5.2 Linking Raw Data

**Do not copy raw files into project folders.** Instead, create references:

**Option A — Symlinks (preferred for local work):**
```bash
ln -s /gjesus3/microscopy/raw/zeiss-wsi/2025/2025-01/20250122_ZWSI_MBC_001/ \
      /gjesus3/microscopy/projects/jrc-lung-fibrosis/inputs/
```

**Option B — Reference file (for portability):**
Create `inputs/sources.txt` listing scan IDs:
```
20250122_ZWSI_MBC_001
20250122_ZWSI_MBC_002
```

### 5.3 Recording Provenance

When creating derived outputs, record the relationship in the provenance ledger:

| output_file | input_scan_ids | process_description | operator | date | software |
|-------------|----------------|---------------------|----------|------|----------|
| `outputs/fig1_panel_a.png` | `20250122_ZWSI_MBC_001` | ROI extraction, brightness adjustment | MBC | 2025-01-22 | ImageJ 1.54 |

This enables trace-back from any publication figure to its raw source(s).

## 6. Registry Specifications

### 6.1 Raw Inventory (`raw_inventory.csv`)

The authoritative record of all raw acquisitions.

| Field | Required | Description |
|-------|----------|-------------|
| `scan_id` | Yes | Unique scan identifier |
| `date_acquired` | Yes | YYYY-MM-DD |
| `instrument` | Yes | Instrument code (e.g., `ZWSI`) |
| `operator` | Yes | Operator code |
| `sample_type` | Yes | Brief sample description |
| `project_code` | No | Associated project (if known) |
| `staining` | No | Staining protocol used |
| `objective` | No | Objective magnification |
| `file_count` | Yes | Number of raw files |
| `total_size_gb` | Yes | Total size in GB |
| `checksum_file` | Yes | Path to checksums.sha256 |
| `notes` | No | Free-text notes |
| `registered_by` | Yes | Who registered the entry |
| `registered_date` | Yes | When registered (YYYY-MM-DD) |

### 6.2 Provenance Ledger (`provenance_ledger.csv`)

Tracks derived outputs and their sources.

| Field | Required | Description |
|-------|----------|-------------|
| `output_id` | Yes | Unique output identifier (auto-generated or path-based) |
| `output_path` | Yes | Path to output file within project |
| `input_scan_ids` | Yes | Comma-separated list of source scan IDs |
| `project_code` | Yes | Project containing this output |
| `process_description` | Yes | What was done to create this output |
| `software` | Yes | Software/version used |
| `operator` | Yes | Who created the output |
| `date_created` | Yes | YYYY-MM-DD |
| `notes` | No | Additional context |

### 6.3 Projects Registry (`projects.csv`)

| Field | Required | Description |
|-------|----------|-------------|
| `project_code` | Yes | Unique project identifier |
| `title` | Yes | Full project title |
| `lead` | Yes | Project lead (operator code) |
| `pi` | Yes | Responsible PI |
| `created_date` | Yes | YYYY-MM-DD |
| `status` | Yes | `active`, `archived`, `completed` |
| `description` | No | Brief description |
| `related_protocols` | No | Links to protocols.io or similar |

## 8. Access Control and Governance

### 8.1 Access Tiers

| Tier | Permissions | Users |
|------|-------------|-------|
| **Admin** | Full read/write to all areas including `_registry/` | Ryan Tasseff, Jesús Ruiz-Cabello |
| **Operator** | Write to `raw/` (deposit only), read/write to `projects/` | All authorized users |
| **Read-only** | Read access to `raw/` and `projects/` | TBD (for external collaborators if needed) |

### 8.2 Compliance Requirements

Access to gjesus3/microscopy is conditional on agreement to:

1. **Follow all naming conventions** exactly as specified.
2. **Register all raw acquisitions** in the raw inventory within 48 hours of acquisition.
3. **Never modify, rename, or delete** files in `raw/` after registration.
4. **Record provenance** for any derived outputs intended for publication or sharing.
5. **Report issues** immediately to the Data Management Lead.

### 8.3 Violation Handling

1. **First violation:** Verbal/written reminder and correction of the issue.
2. **Repeated violations:** Escalation to PI; potential temporary suspension of write access.
3. **Serious violations** (e.g., deletion of raw data): Escalation to PI for review; potential permanent access revocation.

### 8.4 Onboarding Process

New users must:

1. Read this document in full.
2. Attend a 30-minute orientation session with the Data Management Lead.
3. Complete a supervised test deposit (one scan session).
4. Sign acknowledgment of compliance requirements.

## 9. Data Protection Considerations

### 9.1 Standard Data

Most microscopy data (animal tissue, cell lines, phantoms) is **standard sensitivity** and has no special handling requirements beyond the conventions in this document.

### 9.2 Restricted Data (Human/Clinical)

Data derived from human subjects or clinical samples requires additional safeguards:

1. **Mark the project** as `data_sensitivity: restricted` in `_project.yaml`.
2. **No patient identifiers** in file names, folder names, or unencrypted metadata.
3. **Access limited** to named individuals approved for the study.
4. **Tania Orosco Salinas** serves as safeguard/advisor for clinical data handling.
5. **Consult institutional policies** and ethics approvals before data deposit.

## 10. Integrity and Backup

### 10.1 Checksums

All raw acquisitions must have SHA-256 checksums generated at deposit time. Periodic verification (quarterly recommended) ensures data integrity.

### 10.2 Backup Status

**Current state:** gjesus3 is RAID-protected but **not backed up** by IT.

**Mitigation strategies:**

1. **Checksum verification** catches silent corruption.
2. **Original instrument files** may exist on acquisition workstation (short-term).
3. **Critical projects** should maintain an additional copy (external drive or cloud) — responsibility of project lead.

**Future consideration:** Advocate for institutional backup solution as part of platform-level data management.

## 11. OMERO Compatibility

This system is designed for future integration with OMERO (Open Microscopy Environment):

1. **Scan IDs** can serve as OMERO dataset names.
2. **Project codes** align with OMERO project structure.
3. **Raw inventory** metadata can be imported as OMERO key-value annotations.
4. **Provenance ledger** records can inform OMERO ROI and attachment relationships.

When OMERO is deployed, a migration script can ingest registered scans while preserving all metadata and relationships.

## 12. Quick Reference Checklists

### After Every Acquisition

- [ ] Create scan folder with correct naming: `YYYYMMDD_INST_OPR_###`
- [ ] Copy all raw files to the scan folder
- [ ] Generate checksums: `sha256sum *.czi > checksums.sha256`
- [ ] Add README.txt with acquisition notes
- [ ] Add entry to `raw_inventory.csv`

### Starting a New Project

- [ ] Add entry to `projects.csv`
- [ ] Copy project template to new folder
- [ ] Complete `_project.yaml`
- [ ] Create symlinks to relevant raw data in `inputs/`

### Before Publication

- [ ] Verify all figures have provenance ledger entries
- [ ] Confirm raw data checksums still valid
- [ ] Document software versions in provenance records
- [ ] Archive project status to `completed`
